# GNM Studio v1.1.0 implementation plan

## Release objective

GNM Studio 1.1 adds a second, high-fidelity FaceCap mocap avatar, model-aware
materials and exports, stronger face-only head pose, and a clean single-renderer
output mode for fullscreen presentation, OBS window capture, and popout use.

The current GNM Head v3 remains the default avatar. Existing settings, motion
JSON files, and recordings must continue to work.

## Confirmed source-model facts

The avatar used by `face-webgpu.html` is not embedded in that HTML. It loads
Three.js's 332,808-byte `examples/models/gltf/facecap.glb`, credited by the
Three.js example to the creator of the Face Cap app.

The asset contains:

- a morphable head with 52 FaceCap/ARKit-style targets;
- separate left/right eyes and teeth nodes;
- a `grp_transform` pose node;
- normals and UV coordinates on all four meshes; and
- direct targets including `jawOpen`, jaw direction, eye/brow, mouth, cheek,
  nose, and `tongueOut`.

The model is MIT-licensed, as confirmed by the project owner on 2026-07-17. It
was supplied to Three.js by the Face Cap creator and is credited by the Three.js
example to Face Cap. Bundle the model locally for offline desktop/web use and
preserve its MIT status, Face Cap credit, Three.js source URL, and pinned r184
source revision in `THIRD_PARTY_NOTICES.md`.

The HTML's WebGPU, SSGI, TRAA, and Cornell-box renderer are not required for the
model. Version 1.1 should load FaceCap through the existing Three.js renderer so
the Windows WebView2 and wider web fallback keep working. A separate
experimental WebGPU renderer can be considered after 1.1.

## Architecture

### 1. Model profiles and selector

Introduce `AvatarKind = "gnm" | "facecap"` and a model-profile registry. Each
profile owns its asset URL, node names, morph-target map, pose root, material
targets, supported controls, identity capability, and export adapter.

Add a persistent selector in the Avatar panel:

- **GNM Head v3** — 20 semantic expressions, generated jaw target, seeded
  desktop identity generation, and native Rust GNM deformation.
- **FaceCap 52** — direct MediaPipe-to-FaceCap mapping, separate eyes/teeth,
  `tongueOut`, and no GNM seeded identity controls.

Switching models must preserve the camera, calibration, smoothing, background,
lighting, timeline, and capture state. Disable only controls unsupported by the
selected profile and explain why in the UI.

Refactor `Stage` so model loading, face/eye/teeth discovery, normalization,
material assignment, retargeting, and cleanup are profile-driven rather than
hard-coded to the GNM GLB.

### 2. FaceCap mocap and manual controls

Port the complete MediaPipe-to-FaceCap name map from `face-webgpu.html` into a
typed retarget profile. Apply all 52 available values directly instead of
reducing them to GNM's 20 semantic controls.

Add model-aware expression groups for FaceCap:

- eyes and gaze;
- brows;
- cheeks and nose;
- jaw;
- lips and mouth; and
- tongue.

Manual values and per-channel freeze locks must work for both profiles. Reset
must clear only the active model's controls. Calibration and adaptive smoothing
remain upstream of both retarget profiles so they behave consistently.

### 3. Face-only head rotation

Keep MediaPipe's facial transformation matrix as the primary pose source. It is
computed from the face and does not require or follow torso landmarks.

Create a shared `HeadPoseController` that:

- removes the calibrated neutral quaternion;
- converts MediaPipe coordinates once in a documented location;
- applies mirror correction consistently;
- validates yaw/pitch/roll against stable face landmarks such as eye corners,
  nose, forehead, chin, and side-face points;
- falls back to landmark-derived pose if the matrix is absent or spikes;
- rejects implausible one-frame angular jumps;
- exposes head-rotation on/off, yaw/pitch/roll strength, dead-zone, and
  smoothing controls; and
- clamps only extreme angles while preserving deliberate fast turns.

Apply the resulting quaternion to the GNM performance root and FaceCap's
`grp_transform`. Do not add Pose Landmarker or shoulder/torso influence for
head rotation. Record the calibrated quaternion in motion JSON and animate it
in both GLB export profiles.

### 4. PBR skin on both models

Reuse the existing five skin tones, colour/normal/displacement/AO/specular maps,
scale, rotation, and seam-feather controls.

For FaceCap:

- apply skin PBR only to the head/skin mesh;
- retain dedicated neutral materials for eyes and teeth;
- confirm the model's UV orientation and tune its independent texture scale;
- create a skin-region mask or preprocessed material split so inner mouth and
  `tongueOut` geometry never receives pore texture;
- preserve morph targets and meshopt compatibility during preprocessing; and
- keep experimental microtexture OFF by default for both profiles.

The live renderer and exported GLB must use the same material parameters and
mask. Material changes must not flash dark while asynchronous textures reload.

### 5. Model-aware recording and export

The current composited-canvas recorder should work for both avatars once Stage
is profile-driven. Verify Avatar video, Camera + avatar, WebM source, WebCodecs
MP4, and optional system-FFmpeg MP4 with each model.

Extend motion JSON with optional, backward-compatible metadata:

- `avatarKind`;
- retarget-profile/version;
- calibrated face-pose settings; and
- active material settings.

Keep the 52 raw MediaPipe channels and pose matrix as the canonical recording,
so a take can be replayed on either avatar. Warn rather than fail when importing
an older file without model metadata.

Generalize GLB export:

- GNM exports its semantic targets plus generated jaw and root animation;
- FaceCap exports all 52 direct morph tracks, eye transforms, pose-root
  quaternion, separate eyes/teeth, and optional masked PBR skin;
- export names identify the selected avatar; and
- Blender import must preserve timing, head turning, jaw, tongue, and material
  assignment.

When a recorded take is replayed on a different avatar, export what is visibly
selected and include the selected profile in the file metadata.

### 6. Fullscreen clean-output controls

Separate the existing panel-hiding **Focus canvas** mode from real
**Fullscreen output**.

Fullscreen output should:

- use the browser/Tauri fullscreen API;
- show canvas-only output;
- reveal compact controls on pointer movement or keyboard activity;
- fade them after a configurable delay, default 2.5 seconds;
- support `H` to pin/hide controls immediately and `Esc` to exit;
- provide an **Always hide controls** OBS option;
- hide guides, status chips, gizmo, mouse-light control, and cursor when clean
  output is active; and
- respect reduced-motion accessibility settings.

Recording must capture only the composited canvas, never temporary controls.

### 7. Single-renderer popout window

Add **Pop out canvas** beside the viewport controls.

The main application remains the owner of the webcam, microphone, MediaPipe
worker, calibration, timeline, and settings. The popout becomes the sole owner
of the Three.js Stage renderer. Send only tracking controls, pose, settings,
commands, and heartbeat messages through a versioned `BroadcastChannel`
protocol; do not open a second camera or run a second tracker.

Lifecycle:

1. User clicks Pop out canvas.
2. Desktop creates a Tauri `output` WebviewWindow; web uses a same-origin
   user-initiated popup when allowed.
3. Popout loads the selected model and acknowledges renderer readiness.
4. Main Stage unmounts and is replaced with an **Output is in the popout**
   placeholder containing Focus, Bring back, and Close output actions.
5. Video recording commands are executed by the window owning the canvas.
6. If the popout closes, crashes, or misses its heartbeat, the main Stage
   remounts automatically and restores the last view/settings.

Only one Stage/WebGL renderer may be active after handoff. Add diagnostics and
an automated lifecycle test for that invariant. The first 1.1 implementation
should treat popout as clean avatar/background output for OBS Window Capture;
raw webcam compositing stays in the main window unless a later cross-window
video-frame path proves reliable and efficient.

Update Tauri capabilities so both `main` and `output` windows receive only the
permissions they need. The output window must not expose file dialogs,
clipboard, FFmpeg, or unrelated commands.

### 8. Testing and release work

Test both desktop and web editions across:

- GNM and FaceCap model switching;
- camera unavailable/manual-only operation;
- GPU and CPU tracking;
- mirrored and unmirrored yaw/pitch/roll;
- neutral calibration and smoothing;
- all 52 FaceCap targets, especially jaw and tongue;
- skin OFF/ON and every PBR control;
- JSON round-trip and cross-avatar playback;
- GLB import in Blender;
- Avatar and Composite video plus MP4/WebM conversion;
- fullscreen auto-hide and keyboard controls;
- popout open/focus/close/crash recovery;
- recording while the popout owns the canvas; and
- confirmation that only one renderer and one tracker are active.

At feature freeze, bump package, Cargo, Tauri, portable-packaging, badge, and
documentation versions together to `1.1.0`; build standard and UPX portable
ZIPs; update English/Chinese READMEs and third-party notices; run local and
GitHub CI; deploy the web build; then create and push the annotated `v1.1.0`
tag. Do not move the existing `v1.0.0` or `v1.0.1` tags.

## Recommended implementation order

1. Record the confirmed MIT/Face Cap/Three.js attribution and vendor/preprocess the model.
2. Add avatar profile types, selector, and profile-driven Stage loading.
3. Add direct 52-channel FaceCap retargeting and controls.
4. Introduce the shared face-only head-pose controller.
5. Add masked FaceCap PBR material support.
6. Generalize JSON/GLB/video export and test in Blender.
7. Add fullscreen clean-output auto-hide.
8. Add the single-renderer popout transport and recovery lifecycle.
9. Complete regression, performance, documentation, packaging, and release.

## Definition of done

Version 1.1.0 is complete only when both avatars can be selected, tracked,
manually edited, recorded, replayed, exported to MP4/JSON/GLB, and rendered with
skin OFF or masked PBR skin ON; head rotation follows face-only yaw/pitch/roll;
fullscreen clean output leaves no UI in captured pixels; popout handoff never
runs two renderers or trackers; closing the popout restores the main canvas; all
assets have documented redistribution terms; and desktop/web CI passes.
